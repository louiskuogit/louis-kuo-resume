Workflow

從需求到交付的管理流程

1

需求申請

使用者提交帳號、設備、系統權限、雲端資源或資料異動需求。

2

主管簽核

依部門、成本、資料敏感度與權限風險決定是否核准。

3

IT 分派

IT 管理者依類型、優先度與 SLA 指派處理人員與 Runbook。

4

完成驗收

保留處理紀錄、驗收結果與後續改善建議，形成可稽核紀錄。

System Design

可延伸的企業系統架構

真實落地時，前端可接 API 與資料庫，並串接 AD / SSO、郵件通知、工單系統、ERP 權限資料、 稽核紀錄與 BI 報表。Demo 的重點是把流程和資料邊界設計清楚。

Frontend

申請表單、簽核看板、角色視圖、SLA 狀態與報表。

API Layer

流程狀態機、權限檢查、通知、簽核紀錄與資料驗證。

Integration

AD / SSO、ERP、Email、ITSM、監控與資產管理資料。

Governance

稽核軌跡、異動紀錄、角色權限、SLA 與管理報表。

Why This Matters

這個作品能證明什麼

對 HR：不是只有口頭說會管理流程，而是能把流程變成可操作的系統。

對資訊主管：能理解權限、稽核、SLA、Runbook、供應商與跨部門協作。

對 ERP / 企業系統職缺：能把需求訪談、權限控管、資料介接與上線維運串起來。

對 DevOps / SRE 職缺：能把服務穩定度、事件處理、監控與流程改善納入管理節奏。
